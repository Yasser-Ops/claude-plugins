# optimize-images

Raw PNG/JPG screenshots are heavy and inconsistent. Ship WebP, and make every image that fills the **same UI slot** share **one aspect ratio** so the layout crops them identically.

**Core rule:** the display box and the source image must share the same aspect ratio. When they match, `object-cover` crops nothing — the "edges are cut off" problem disappears.

## When to use

- New screenshots/photos land in `public/` (covers, inline shots, avatars).
- Covers/thumbnails look cropped, letterboxed, or inconsistent with each other (especially on mobile).
- A raw `.png`/`.jpg` is referenced in code instead of `.webp`.

## The pipeline

Use `sharp` (add as a devDependency: `pnpm add -D sharp`). Keep a repeatable script at `scripts/optimize-images.mjs` wired to `package.json` as `"optimize:images": "node scripts/optimize-images.mjs"`. Re-run it whenever a source image changes.

Group images by **role**, then process each role consistently:

| Role | Treatment |
| --- | --- |
| Covers / same-slot thumbnails | Crop to ONE shared ratio (e.g. `2:1`), anchored `top`, capped width (≤1600px) |
| Inline content shots | Preserve native ratio, cap width (≤1600px) |
| Avatars / portraits | Crop to the display box ratio at 2x for retina |

```js
// scripts/optimize-images.mjs — adapt the tasks list per project.
import sharp from "sharp";
import path from "node:path";
import { fileURLToPath } from "node:url";

const root = path.resolve(path.dirname(fileURLToPath(import.meta.url)), "..");
const p = (rel) => path.join(root, "public", rel);
const MAX_WIDTH = 1600;

// kind: "cover-2x1" (uniform 2:1, top-anchored) | "width" (cap width) | custom
const tasks = [
  { in: "shots/a/cover.png", out: "shots/a/cover.webp", kind: "cover-2x1" },
  { in: "shots/b/cover.png", out: "shots/b/cover.webp", kind: "cover-2x1" },
  { in: "shots/a/inline.png", out: "shots/a/inline.webp", kind: "width" },
];

for (const t of tasks) {
  let img = sharp(p(t.in));
  if (t.kind === "cover-2x1") {
    img = img.resize(MAX_WIDTH, MAX_WIDTH / 2, { fit: "cover", position: "top" });
  } else if (t.kind === "width") {
    const { width = 0 } = await img.metadata();
    if (width > MAX_WIDTH) img = img.resize(MAX_WIDTH);
  }
  const info = await img.webp({ quality: 80, effort: 6 }).toFile(p(t.out));
  console.log(`${t.out}  ${info.width}x${info.height}  ${(info.size / 1024).toFixed(0)}KB`);
}
```

## Steps

1. Inventory `public/` images and find every reference (`grep` for the filenames in `src/`, `.mdx`, manifests).
2. Group by role; pick the shared target ratio for each same-slot group.
3. Run the sharp pipeline → `.webp` outputs.
4. **Match the frame to the ratio.** Set the display box to the same aspect (e.g. `aspect-[2/1]`) so `object-cover` does zero extra cropping. Use `object-top` so any unavoidable crop keeps the top of the shot.
5. Update all references from `.png`/`.jpg` to `.webp`. Unify filenames across the group (e.g. every cover is `cover.webp`) so per-slug branches collapse.
6. Delete the original PNG/JPG sources (`git rm`); keep the `.webp`.
7. `pnpm typecheck && pnpm lint && pnpm build`.

## Leave alone

- Favicons / app icons (`src/app/icon.png`, `apple-icon.png`) — small PNGs Next.js handles natively; WebP is not appropriate for favicons.
- `*.svg` — already optimal.
- OG images generated by `opengraph-image.tsx` — handled by Next at build.

## Common mistakes to avoid

- Setting the display box to a different ratio than the unified source — reintroduces edge cropping.
- Unifying to a ratio that crops the source's sides (e.g. forcing a wide 2:1 shot into 16:10) when the goal was to *stop* cropping — pick the ratio that the sources already sit near.
- Converting favicons/app icons to WebP, or leaving raw PNGs referenced after conversion.
- Forgetting to delete the originals — they bloat the build even when unreferenced.
- Skipping `pnpm build` — stale references only surface at build/runtime.
